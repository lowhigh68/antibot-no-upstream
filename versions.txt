## [4.4.9] — 2026-05-24 +07:00

### Changed / Performance
- **Fix A — body scan scoped to `application/x-www-form-urlencoded` only** (`core/req_classifier.lua`).
  Root cause: `body_contains_auth_marker()` gọi `ngx.req.read_body()` cho MỌI POST không match
  keyword/legacy/qs path — kể cả REST/JSON (WooCommerce API, SPA AJAX), GraphQL, multipart upload.
  Cost ~10-50µs mỗi call, trên 150 site / 100 CCU với nhiều JSON POST → aggregate latency đáng kể.

  Fix: thêm Content-Type guard đầu `body_contains_auth_marker(ct)` — nếu CT không phải
  `application/x-www-form-urlencoded` → return false ngay, zero `read_body`.
  - JSON POST: REST API, WooCommerce add-to-cart/checkout, SPA → zero overhead (caught by keyword path anyway)
  - multipart POST: file upload → zero overhead (never a credential vector)
  - form-urlencoded POST: browser login form, credential submit → body scanned as before
  Cleanup: `AUTH_BODY_MARKERS` loại bỏ JSON/multipart markers (orphaned by Fix A — chúng không
  bao giờ được check nữa). List giảm từ 19 → 11 entries, chỉ giữ form-urlencoded markers.

- **Fix B — mở rộng `AUTH_LEGACY_PATHS` cho multi-CMS server** (`core/req_classifier.lua`).
  Root cause: CMS admin backends ngoài WordPress không có auth semantic keyword trong URI và không
  dùng standard form-urlencoded credential fields → classify là `interaction` (rate_weight=0.3)
  → admin AJAX burst không bị throttle.

  Changes:
  - `"^/wp%-admin/admin%-ajax%.php$"` → mở rộng thành `"^/wp%-admin/"` (tất cả WP admin POST:
    heartbeat, autosave, plugin AJAX, form saves — không chỉ admin-ajax.php).
  - `"^/administrator/"` — Joomla admin dispatcher (global #2 CMS, phổ biến VN government sites).
    Tất cả Joomla admin ops POST qua /administrator/, targeted cho com_users credential attack.
  - `"^/filament/"` — Laravel Filament admin panel (growing VN Laravel ecosystem).
  - `"^/nova/"` — Laravel Nova admin panel.
  Sau fix: tất cả paths trên → `auth_endpoint` (rate_weight=1.5) → throttle admin burst.
  Note: Drupal `/admin/`, OpenCart `/admin/`, NukeViet `/admin/` không thêm vì path quá broad
  trên multi-site server (FP risk). Login page của các CMS này đã caught bởi semantic keyword.

### Files changed
- antibot-core/core/req_classifier.lua  (Fix A: body scan CT guard + markers cleanup; Fix B: AUTH_LEGACY_PATHS expansion)

## [4.4.8] — 2026-05-24 +07:00

### Fixed
- **503 Service Unavailable trên toàn bộ WordPress sites** sau khi deploy v4.4.4+v4.4.5.
  Hai nguyên nhân cộng hưởng:

  A) `inapp_browser rate_weight` 0.3 → 0.6.
     v4.4.5 classify mobile in-app requests (Zalo/FB) vào class mới `inapp_browser`
     với rate_weight=0.3 — chỉ 30% so với navigation (1.0). Kết quả: 3× nhiều traffic
     hơn pass qua Apache/PHP so với trước. Trên các site VN có lượng Zalo in-app
     cao → Apache workers tăng → PHP memory pool cạn → 503.

  B) `/wp-admin/admin-ajax.php` thêm lại vào `AUTH_LEGACY_PATHS`.
     v4.4.4 xóa admin-ajax.php khỏi AUTH_LEGACY_PATHS (lý do: body detection catch
     brute force action=login). Side effect: endpoint bị reclassify từ
     `auth_endpoint` (rate_weight=1.5) → `interaction` (rate_weight=0.3) vì URI
     không có auth keyword và body heartbeat/autosave không chứa credential marker.
     WordPress Heartbeat API (mỗi 15s/session) + autosave + WooCommerce admin ops
     burst qua Apache không bị rate limit → overload.

### Files changed
- antibot-core/core/req_classifier.lua  (inapp_browser rate_weight + AUTH_LEGACY_PATHS)

## [4.4.7] — 2026-05-24 +07:00

### Fixed
- **FP: Googlebot smartphone UA blocked as inapp_browser on www.hoanmy.vn** (IPs 66.249.66.40, 66.249.71.132).
  Two root causes:

  A) `compute_inapp_likeness()` — Signal 2 (non-canonical Safari tail) + Signal 3 (Chrome-missing-Sec-Ch-Ua)
     fire on Google bot UAs (inapp_likeness=0.45 → classified as `inapp_browser`):
       `...Mobile Safari/537.36 (compatible; Googlebot/2.1; +http://...)`
       `...Mobile Safari/537.36 (compatible; GoogleOther)`
     Crawlers intentionally append RFC 9309 tokens after `Safari/X.Y`. In-app browsers never do this.
     Fix: two-part guard in `compute_inapp_likeness`:
       (a) name token: `bot`/`spider`/`crawler` — catches Googlebot, Bingbot, etc.
       (b) RFC 9309 compatible-comment at UA tail `%(compatible;[^%)]+%)%s*$` —
           catches GoogleOther and any future compliant bot without "bot" in name.
     Confirmed from actual access log: both IPs use BOTH UA forms.

  B) `ua_check.lua` unregistered_bot path — when Redis doesn't have `goodbot:dns:<name>` (seed
     not yet run, transient flush), `ctx.good_bot_asns` was not set. ASN fallback in `bot/init.lua`
     checks `ctx.good_bot_asns` → nil → can't fallback to hardcoded AS15169.
     Fix: add `ctx.good_bot_asns = get_bot_asns(bot_name)` in unregistered_bot path so
     hardcoded PTR_ONLY_BOTS ASNs are always available as fallback regardless of Redis state.

### Files changed
- antibot-core/core/req_classifier.lua  (two-part bot-exclusion guard in compute_inapp_likeness)
- antibot-core/detection/bot/ua_check.lua  (unregistered_bot path: set ctx.good_bot_asns)

## [4.4.5] — 2026-05-24 +07:00

### Changed
- core/req_classifier.lua — **inapp_browser: generic structural detection, xoá brand-token enumeration**.
  - Xoá `INAPP_UA_TOKENS` (FBAN, ZALO, INSTAPP, ...) và `is_inapp_browser()`. Same anti-pattern:
    app mới → thêm token = dead-end maintenance.
  - Thay bằng `compute_inapp_likeness(ua, xrw, sec_ch_ua)` → 3 structural signals:
    * Signal 1 (0.6): X-Requested-With = reverse-domain bundle ID (Android WebView tự set).
      Covers ALL Android WebView apps kể cả chưa ra đời. Standalone browser không set header này.
    * Signal 2 (0.3): UA không kết thúc canonical `Safari/X.Y`. WebView/in-app append token
      sau Safari hoặc bỏ hẳn (iOS Facebook: `... Mobile/15E148 [FBAN/FBIOS;...]`).
    * Signal 3 (0.15): Chrome 90+ UA nhưng thiếu `Sec-Ch-Ua` — WebView không expose client hints.
    * OS-engine safety net (0.3): OpenHarmony, HarmonyOS, HMSCore, ArkWeb, TBS/, KAIOS.
  - Output: `ctx.inapp_likeness ∈ [0,1]`. Threshold 0.4 → class=`inapp_browser`.
  - async/logger.lua: thêm `inapp=%.2f` vào antibot.log format.

### Rationale
Generic catches MỌI app hiện tại + tương lai (Android/iOS WebView, HarmonyOS NEXT). Zero code
change khi app mới ra. OS-engine list thay đổi 1-2 lần/decade — acceptable maintenance.

### Files changed
- antibot-core/core/req_classifier.lua  (replace INAPP_UA_TOKENS → compute_inapp_likeness/detect_inapp)
- antibot-core/async/logger.lua         (thêm inapp=%.2f)

## [4.4.4] — 2026-05-24 +07:00

### Changed
- core/req_classifier.lua — **auth_endpoint v3-hybrid: keyword fast path + body semantic fallback**.
  - v1: enumeration ~30 CMS hardcoded patterns — anti-pattern, removed.
  - v2: `AUTH_KEYWORDS` semantic vocabulary (login, signin, auth, oauth, password, token, 2fa...).
    `has_auth_keyword_in_path()` split URI theo "/", word-boundary check `[-_.]`. Catches
    `/wp-login.php`, `/user/login`, `/customer/account/login`, `/api/v1/auth/login` via single
    keyword, zero CMS-specific code. `has_auth_keyword_in_args()` cho Joomla/OpenCart query routing.
  - v3 body fallback (NEW): `ngx.req.read_body()` + scan `AUTH_BODY_MARKERS` (password=, passwd=,
    "password", name="password", client_secret=, grant_type=password, otp=, mfa_code=...).
    Body cap 8KB. Defense-in-depth: attacker phải bypass URI keyword AND body marker.
  - `AUTH_LEGACY_PATHS` giảm 2→1 entry (`/xmlrpc.php` only — XML body không có `password=` literal).
    `/wp-admin/admin-ajax.php` removed — body detection catches `pwd=` / `password=`.
  - Order fix: auth check trước interaction JSON (tránh `/wp-json/wp/v2/users` bị classify nhầm).

### Rationale
v3 catches obfuscated paths (Magento `loginpost`, `/portal/enter`, `/api/v9/handshake`) khi body
chứa credential field. Fast path (95% traffic) zero overhead. Slow path ~10-50µs chỉ trigger POST.

### Files changed
- antibot-core/core/req_classifier.lua

## [4.3.6] — 2026-05-16 10:45:18 +07:00

### Added
- async/logger.lua — Beacon coverage telemetry (Step 0 baseline metric).
  - New log field `beacon=1|0|skip` appended to each line in `/var/log/antibot/antibot.log`.
    * `1`    = HTML-eligible request AND beacon data available in Redis (canvas/webgl signal analyzable).
    * `0`    = HTML-eligible request BUT beacon not yet collected (first visit / JS blocked / CSP fail).
    * `skip` = non-HTML class (resource/api/auth) — beacon not applicable.
  - New Redis daily counters (TTL 7d):
    * `stat:<host>:beacon_elig:<YYYYMMDD>` — count of HTML-eligible requests
    * `stat:<host>:beacon_got:<YYYYMMDD>`  — count of those with beacon data available
  - Coverage = `beacon_got / beacon_elig` per host per day.

### Rationale
Prerequisite for upcoming canvas.lua / webgl.lua signal implementation. Without coverage
baseline, weight tuning for the new signals would be guesswork. Coverage below 30% means
the JS inject path itself is broken upstream and new signals would be ineffective for
majority of requests.

### Risk
- Zero functional impact on request decisioning (telemetry-only).
- Negligible perf: +2 Redis INCR per HTML-eligible request (already inside existing
  `write_stats` pipeline → no new round trip).
- Log line +12 bytes on average.

### Files changed
- antibot-core/async/logger.lua
  * write_stats() signature: added `beacon_got` parameter
  * write_stats() body: added 2 Redis counters block
  * run() body: added `beacon_state` computation for log line
  * run() body: added `beacon=%s` field in log format string + arg list
  * run() body: pass `beacon_got` to write_stats via timer.at

### Deployment
1. Copy file to /usr/local/openresty/nginx/conf/antibot/async/logger.lua
2. /usr/local/openresty/nginx/sbin/nginx -t && /usr/local/openresty/nginx/sbin/nginx -s reload
3. Wait 24-48h for traffic accumulation
4. Read coverage via:
   - Log:   awk '/beacon=[01]/ { if (match($0,/beacon=1/)) g++; else m++ } END { print g, m, g*100/(g+m) }' /var/log/antibot/antibot.log
   - Redis: redis-cli GET stat:<host>:beacon_elig:<YYYYMMDD> ; GET stat:<host>:beacon_got:<YYYYMMDD>

### Next planned (not in this version)
- canvas.lua implementation — analyze ctx.browser.cv for empty/known-headless/suspicious patterns
- webgl.lua implementation — analyze ctx.browser.wgl for SwiftShader/Mesa/UA-OS mismatch
- Weights tuning in intelligence/scoring/compute.lua based on coverage observed

## [4.3.5] — 2026-05-16 (earlier today, pre-release)

### Added
- nginx/da_to_openresty.sh + nginx/da_to_openresty-running.sh — X-Forwarded-Port header.
  - Added `proxy_set_header X-Forwarded-Port $server_port;` to both HTTP and HTTPS proxy blocks
    in both template files.
  - Total: 4 insertions (2 files × 2 blocks each).

### Fixed
- "Your request is blocked" on POST/admin login for NukeViet-based sites behind OpenResty.
  - Root cause: NukeViet vendor/vinades/nukeviet/Core/Server.php constructor's reverse-proxy
    override block activates when BOTH X-Forwarded-Host AND X-Forwarded-Proto are present.
    When X-Forwarded-Port is absent, fallback uses pre-formatted `server_port` string
    (':8081' or ''), then loose-compare `!= 80 and != 443` evaluates true for empty/prefixed
    string → prepends another ':' → trailing-colon URL like 'https://cntm.edu.vn:'.
  - Origin from browser is clean 'https://cntm.edu.vn' → string mismatch → REQUEST_BLOCKED
    at Request.php:451.
  - Sending X-Forwarded-Port=443 (or 80) gives NukeViet a clean numeric string that PHP
    loose-compares correctly against 443 → falls into else branch → no extra colon prefix.

### Removed
- nginx/da_to_openresty-running.sh — Dead code declarations.
  - Removed `OR_HTTP_PORT="17080"` and `OR_HTTPS_PORT="17443"` (lines 41-42).
  - These vars were never referenced anywhere in the file — orphan stale declarations from
    earlier port-mapping experiment that never landed. Align with da_to_openresty.sh which
    never had them.

### Affected CMSs
- NukeViet 4.x — primary target, confirmed fix on cntm.edu.vn
- WordPress, Joomla, Drupal, Laravel, Symfony — receive correct port info (no-op if app
  doesn't read X-Forwarded-Port, otherwise produces correct canonical URLs)
- Static / non-PHP — no-op

### Risk assessment
- Security: Zero — `$server_port` is nginx variable (server listen port), not user-controlled.
- Backward compat: 100% — header is additive, ignored by apps that don't read it.
- Antibot/FP impact: Net positive — fewer cross-site/origin false positives because apps
  build canonical URLs correctly.

### Files changed
- nginx/da_to_openresty.sh
- nginx/da_to_openresty-running.sh
