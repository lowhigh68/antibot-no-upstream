2026-05-23 — Generic FP fixes: unknown-class dampening + res_ip gating
  Problem: tuart.xuongweb.com (WordPress + Flatsome theme install) blocked
  legitimate admin user 14.232.154.94. Two compounding root causes:

    A) class=unknown score_multiplier=1.0 — uncertainty principle ngược:
       request không classify được nhận FULL penalty thay vì giảm. Path
       như /wp-admin/themes.php (GET), Joomla /administrator, Drupal /?q=admin
       rơi vào unknown → score x1.0 → vượt block threshold dễ.

    B) resource_starved (weight=30) đếm res_count theo identity. Nhưng
       resource class skip fingerprint → ctx.identity=nil → res_count per
       identity LUÔN = 0 cho mọi browser session. Browser thật load CSS/JS
       hàng chục lần nhưng signal vẫn fire vì counter bị tách bởi class.

  Fix A' — Uncertainty dampening (generic, mọi CMS):
    core/req_classifier.lua: unknown class score_multiplier 1.0 → 0.5,
    rate_weight 1.0 → 0.5. Nguyên tắc bayesian: uncertainty GIẢM penalty,
    không tăng. Bot thật imitate navigation/interaction pattern → đã rơi
    vào class cụ thể; edge-case URL thường là admin panel legitimate.
    Signal khác (anomaly/h2_bot/cluster) vẫn fire độc lập, đủ escalate
    nếu thực sự bot.

  Fix B' — Resource counting by IP (semantic fix):
    + l7/rate/res_ip_counter.lua (NEW): chạy trong STEPS_RESOURCE, INCR
      res_ip:<ip> với TTL=ttl.rate (60s). Không cần identity.
    + init.lua: thêm res_ip_counter đầu STEPS_RESOURCE.
    + detection/session/session_store.lua: trước khi fire resource_starved,
      đọc res_ip:<ip>. Nếu >= 5 → skip (IP đang load resource, signal sai
      semantic). Threshold 5 chọn để 1 stray hit từ user khác trên NAT
      không giải vây cho bot (giảm FN risk shared-NAT).

      thắng, "resource_starved fp=... res_ip=X" khi fire thật.

2026-05-22 — l7/rate ip_surge hybrid (S2.6)
  Problem: l7/rate/adaptive_limit.lua hard-banned ip on ip_rate>1500/60s
  (~25 req/s) with 1800s TTL, blind to identity diversity and overall score.
  Single user installing/testing a browser extension (Claude Code observed,
  but pattern applies to any extension that prefetches or AI-agent that
  navigates) bursts 1000-1500 requests during first 60s — exceeded threshold,
  whole public IP (CGNAT) banned for 30 min. Affected user's NAT peers too.
  Fix (hybrid model):
    - ip_rate > cfg.rate.ip_surge_threshold (1500) → ctx.ip_surge=true signal
      only. Contributes weight=25 in intelligence/scoring/compute.lua →
      reaches MONITOR (25) alone; needs other signals to escalate.
    - ip_rate > cfg.rate.ip_surge_extreme (5000, ~83 req/s) AND distinct
      identities seen in window < cfg.rate.ip_surge_distinct_min (3) → hard
      ban with cfg.rate.ip_surge_ban_ttl (300s, was 1800).
    - core/redis_pool.lua: new safe_scard() helper.
    - l7/rate/counter.lua: SADD rate:ids:<ip> per request (TTL=ttl.rate) so
      adaptive_limit can compute distinct-identity diversity cheaply.
    - intelligence/scoring/compute.lua: register ip_surge signal + weight 25.
