2026-05-24 (v3-hybrid) — auth_endpoint: thêm body semantic fallback
  v2 (commit 6298a44) dùng semantic keyword path detection. Vẫn còn gap:
  CMS đặt tên endpoint unconventional (Magento `loginpost` component
  không separator, custom obfuscated paths /portal/enter, /api/v9/xyz)
  bị miss. Keyword detection assume framework dùng semantic naming —
  vẫn là assumption, không phải invariant tuyệt đối.

  Tư duy ngược lại — login action có gì THỰC SỰ invariant?
    A login attempt = POST + body chứa credential field (password/OTP).
    Path/URL chỉ là routing — KHÔNG semantic của hành động đăng nhập.
    BODY mới là evidence vật lý của credential transmission.

  v3-hybrid: keyword fast path + body semantic fallback
    Fast path (existing v2):
      1. URI path keyword (AUTH_KEYWORDS)
      2. Legacy path (xmlrpc.php only — XML body không có literal password=)
      3. Query string keyword (Joomla/OpenCart routing)
    Slow path (NEW):
      4. ngx.req.read_body() + scan body cho AUTH_BODY_MARKERS:
         password=, passwd=, pwd=, passphrase=  (form-urlencoded)
         "password", "passwd", "pwd", "passphrase"  (JSON)
         name="password", name="pwd"  (multipart)
         client_secret=, "client_secret"  (OAuth)
         grant_type=password, "grant_type":"password"  (OAuth ROPC)
         otp=, totp=, mfa_code=, mfa_token=, verification_code=  (2FA)

    Defense-in-depth: 95% requests catch ở fast path (zero body read).
    5% obfuscated/custom catch ở slow path. Attacker phải bypass cả hai
    — đổi tên endpoint + đổi tên password field (phá UX).

  AUTH_LEGACY_PATHS giảm từ 2 → 1 entry (/xmlrpc.php only).
  /wp-admin/admin-ajax.php removed — body detection catches action=login
  qua marker pwd= hoặc password=.

  Performance:
    Fast path: O(1) string find trên URI ngắn. Negligible.
    Slow path: chỉ trigger cho POST không match fast path → minority of
    traffic. ~10-50µs per body read. Site 100 req/s với 10% POST nhưng
    8% match fast path → ~2 req/s body read → 0.1ms aggregate latency.

  Coverage delta (case không catch bởi v2 keyword):
    Magento /customer/account/loginpost POST password=...    → catch ✓
    Custom /portal/enter POST password=...                    → catch ✓
    Custom /api/v9/handshake POST "password":...              → catch ✓
    GraphQL /graphql POST với mutation Login                   → catch nếu body có "password"

    antibot-core/core/req_classifier.lua

  Trade-off chấp nhận:
    + Pro: zero maintenance khi CMS mới ra; consistent với principle

2026-05-23 (v3) — Generic kill-switch cho dampened class non-resource
  Problem: Fix A' (commit 649031e — unknown mult 1.0 → 0.5) tạo ra FN.
  Incident observed in log:
    domain=www.bachhoathaomoc.com class=unknown ip=20.9.70.139 (Azure
    AS8075 Microsoft cloud), ua=EMPTY, tls13=nil h2=false, raw_score=140
    + multipliers: bot_score 22%, anomaly 17%, asn_rep 13%, resource_starved
    25%, ua_flag 12%. CLEARLY bot — UA empty là red flag tối đa.

    NHƯNG eff = raw × mult = 140 × 0.5 = 70 → action=challenge MÃI MÃI,
    không bao giờ vượt block threshold (80). Bot không solve PoW → fail
    challenge → request mới lại challenge → loop vô hạn. Detection ĐÚNG
    nhưng enforcement BẤT LỰC do class dampening "che chở" bot.

  Root cause architectural: dampening class_multiplier (resource 0.2,
  interaction 0.6, api_callback 0.5, unknown 0.5, ...) designed để giảm
  FP cho normal traffic, KHÔNG để bảo vệ pure-threat signal storm. Khi
  raw score cực cao (multiple bot signals fire đồng thời), dampening đè
  effective xuống dưới block threshold → bot lách.

  Resource class đã có kill-switch riêng từ trước (KILL_BLOCK_RAW=95→
  EFF=85, KILL_CHALLENGE_RAW=80→EFF=60). Pattern này cần áp dụng generic
  cho mọi class dampened — không phải chỉ resource.

  Fix — generic kill-switch trong enforcement/decision/engine.lua:
    Constants mới:
      KILL_DAMP_HARD_RAW = 150  → floor 85% raw  ("kill_damp_hard")
      KILL_DAMP_SOFT_RAW = 110  → floor 65% raw  ("kill_damp_soft")

    Logic chèn vào nhánh `elseif multiplier < 1.0 then` sau resource block:
      if raw >= 150 then
          eff = max(eff, raw × 0.85)
      elseif raw >= 110 then
          eff = max(eff, raw × 0.65)
      end

    Áp dụng cho: interaction (0.6), api_callback (0.5), feed_or_meta (0.4),
    inapp_browser (0.4), unknown (0.5). KHÔNG áp dụng cho resource (đã có
    kill riêng) và navigation (mult=1.0, không dampened).

  Verification case 20.9.70.139:
    raw=140, mult=0.5 (unknown), eff trước fp_penalty = 70.
    140 >= 110 → soft kill → floor = 140×0.65 = 91 → eff = max(70, 91) = 91.
    91 >= 80 → action=block ✓
    kill_reason = "kill_damp_soft" → logged in error.log + antibot.log.

  FP analysis (chống over-kill):
    Threshold raw 110 chọn cao vì:
      - Logged-in admin user có session_richness 0.7-0.9 → -21 đến -27 pts
        trừ vào raw → "raw" thực tế đã account cho trust.
      - Để đạt raw 110 SAU khi trừ trust, signals threat thực tế phải
        ~135 pts → multiple bot signals fire đồng thời.
      - Legit user load AJAX heavy bị mismatch + h2_bot_confidence khi
        H2 coalescing chỉ contribute ~30-50 pts. Không đạt 110.
      - Resource_starved signal đã có IP-level gating (commit trước) → không
        fire cho legit browser.

    FN coverage cho bot:

        escalate. Acceptable.
      - Bot raw 110-149: soft kill, eff = 71-97 → challenge/block.
      - Bot raw ≥ 150: hard kill, eff ≥ 127 → block chắc chắn.

  Problem: burst handling chỉ có 1 ngưỡng cứng (30/1s/identity) cho mọi class
  + mọi client. Hai dimension không được phân biệt:

    1) Client state — logged-in admin có session với server vs anonymous
       first-visit. Mỗi CMS dùng cookie schema riêng (WP wp_logged_in_*,
       Joomla, Drupal SESS<hash>, Magento admin/frontend, SPA JWT trong
       Authorization header). Hardcode list = dead end maintenance.

    2) Request nature — AJAX-heavy SPA frontend (Magento Luma, Shopify,
       headless commerce) fire 30-50 XHR đồng thời trên page load. Nav
       request thì human không thể burst > 20/s. Một threshold không
       fit cả hai.

  Fix architecture:
    + core/session_richness.lua (NEW): ctx.session_richness ∈ [0,1].
      Generic proxy — measure BẰNG CHỨNG VẬT LÝ rằng client có state:
        - Cookie payload bytes (SIZE_SATURATION 500)
        - Cookie count (COUNT_SATURATION 4)
        - Authorization header present (+0.3 bonus)
        - X-CSRF-Token / X-XSRF-TOKEN present (+0.2 bonus)
      Hoạt động cho MỌI CMS hiện tại + tương lai — không hardcode tên.
      Run trong STEPS_COMMON (sau ctx_layer.init).

    + core/config.lua: rate.class_burst_factor table.
      Multiplier theo class nature:
        navigation 0.67   → 20/s (tighten — F5 storm + multi-tab realistic max)
        interaction 1.5  → 45/s (relax — SPA legit AJAX burst)
        api_callback 2.0 → 60/s (relax — webhook retry storm)
        auth_endpoint 0.8 → 24/s (tighten — credential stuffing 1s hammer)
        feed_or_meta 0.5 → 15/s (tighten — crawler 1 fetch/identity)
        inapp_browser/unknown 1.0 → baseline

    + l7/burst/burst_decision.lua: rewrite threshold logic.
      effective = base × class_burst_factor[class] × (1 + richness × 2)
      Hai dimension orthogonal × multiplicative combine.

    + l7/rate/adaptive_limit.lua: apply session_richness lift vào per-ID
      rate threshold. KHÔNG apply vào ip_surge (per-IP) vì NAT 1 IP có
      nhiều session — richness request hiện tại không đại diện cả IP.

    + intelligence/scoring/compute.lua: session_richness là NEGATIVE

    + async/logger.lua: append richness=X.XX field vào antibot.log mỗi
